# cc-fasrc

Run Claude Code **perpetually on a FASRC login node**, with writes confined to
your lab directory — while `sbatch`, `squeue`, `uv`, and `node` all work normally.

```bash
ssh you@login.rc.fas.harvard.edu
git clone https://github.com/heyodog0/cc-fasrc.git ~/code/cc-fasrc
cd ~/code/cc-fasrc
CC_SANDBOX_DIR=/n/holylabs/<your_lab>/Users/$USER CC_REMOTE_CONTROL=1 ./install.sh
cc-doctor      # verify everything on this node
cc-up          # start (or reattach) the perpetual session
```

## How it confines writes — two modes, auto-detected

`install.sh` picks the strongest enforcement your node supports:

**Native mode (default when available) — kernel-enforced.** Uses Claude Code's
built-in **bubblewrap sandbox**: bash can only write under `$CC_SANDBOX_DIR`,
`/tmp`, and the package caches; everything else is a **read-only filesystem** at
the OS level — not a heuristic. `sbatch`/`squeue`/`srun`/… are in
`sandbox.excludedCommands` so Slurm runs unsandboxed (full munge/network), and
`network.allowedDomains:["*"]` leaves egress unrestricted so git/pip/npm work.
`failIfUnavailable:true` makes `cc` refuse to start rather than run unconfined.

> Needs `bwrap` + `socat` + working user namespaces. FASRC has bwrap and userns;
> `socat` you build once into `~/.local` (5 lines — see *Installing socat*).

**Hook mode (automatic fallback) — no dependencies.** A pure-Python `PreToolUse`
guard (`guard-write.py`): hard-blocks `Write`/`Edit` outside the tree, best-effort
source/dest-aware blocking on `Bash` (`rm -rf /`, redirects, `mv`/`cp` out). Used
on any box without bwrap/socat. Weaker (a contrived shell line could slip the Bash
heuristic) but fine for the "don't let CC fat-finger something" threat model.

Force either with `CC_FORCE_MODE=native|hook ./install.sh`; `cc-doctor` reports
which is active.

### Installing socat (to get native mode on FASRC)

```bash
cd /tmp && curl -fsSLO http://www.dest-unreach.org/socat/download/socat-1.7.4.4.tar.gz
tar xf socat-1.7.4.4.tar.gz && cd socat-1.7.4.4
./configure --prefix="$HOME/.local" --disable-openssl --disable-readline
make -j4 && make install      # lands socat in ~/.local/bin
```
Then re-run `install.sh` and it switches to native mode.

## Why it's useful (vs. just settings.json)

Most of cc-fasrc isn't the guard — it's the operational layer for running CC
perpetually on a cluster, none of which comes from a config file:

- **No results bottleneck.** Jobs write `outputs/`/`logs/` into the same lab tree
  CC sits in, so CC reads progress live — no rsync, no W&B round-trip.
- **Perpetual + reachable.** A `tmux` that survives disconnects; reattach from any
  device. Starts in `auto` permission mode with Remote Control active.
- **Bash path-confinement that config can't express.** Claude Code's `deny` rules
  don't apply to subprocess writes (`echo > file`, a python script) — only the
  native sandbox or a hook can confine those. This is the hook.

## Commands

```
cc-up [name]     start or reattach the session (default name: cc)
ccd  [name]      start detached and return immediately (launch-and-leave)
cck  [name]      kill the session
cc-doctor        preflight: python3, tmux, claude, sbatch, settings, guard, auth
cc               raw sandboxed launcher (cc-up/ccd call this; aliased from `claude`)
```

Inside tmux: `Ctrl-b d` detach · `Ctrl-b :kill-session` kill.

## Two environments (important)

- **CC's session** — login node, write-confined. CC edits code and submits jobs.
- **The SLURM job** — compute node, *outside* the sandbox, full GPU/CUDA access.
  That's correct: you never wanted to sandbox the training run.

So **bootstrap your toolchain as yourself, once** (install node/uv, clone repos) —
let a human run that, not CC. Keep CC a coordinator that `sbatch`es and polls;
heavy compute goes out as jobs, not on the login node.

## Login-node pinning

FASRC round-robins `login.rc...` across `boslogin0N` / `holylogin0N`; your tmux
lives on exactly one. `cc-up`/`ccd` print the host and save the reconnect command
to `~/.cc-fasrc-host`. Reconnect with `ssh you@<that-host>.rc.fas.harvard.edu`,
then `cc-up`.

## Headless auth

So CC survives reboots without a browser:

```bash
umask 077; printf %s 'sk-ant-...' > "$CC_SANDBOX_DIR/.cc/.claude/api-key"
```

Or run `cc` once and complete the OAuth URL by hand — the token persists.

## Files

```
install.sh                       one-time setup (idempotent, auto-detects mode)
config/settings.native.json.tmpl native bubblewrap-sandbox config (default)
config/settings.hook.json.tmpl   hook-mode config (fallback)
hooks/guard-write.py             the hook-mode write guard (run --selftest to verify)
bin/cc                           launcher (sets CLAUDE_CONFIG_DIR + auto mode)
bin/cc-up bin/ccd bin/cck        tmux lifecycle
bin/cc-doctor                    mode-aware preflight verifier
config.env                       generated by install.sh — CC_SANDBOX_DIR, CC_FASRC_MODE
```
